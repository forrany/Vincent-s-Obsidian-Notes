AG-UI 协议核心概念

开放、轻量级协议，连接 AI Agent 后端与前端应用，实现实时同步和交互。

前端技术栈选择 (快速原型)

  • 核心考量: 开发速度、现有资源、实时数据流处理能力

React 优势

  • 生态成熟，第三方库丰富
  • AG-UI 生态集成 (CopilotKit React 组件)
  • 灵活的状态管理 (Hook, Redux, Zustand, Immer)
  • 强大社区支持

React 劣势

  • 学习曲线
  • 样板代码 (相对 Vue)

Vue 优势

  • 易于上手，直观模板语法
  • 响应式系统高效处理状态变化

Vue 劣势

  • AG-UI 特定组件较少
  • 特定领域生态成熟度 (AI Agent 交互)

Next.js 优势

  • 全栈能力 (API 路由、SSR)
  • 优秀开发体验
  • 性能优化 (基于 React)

Next.js 劣势

  • 复杂性增加
  • 学习曲线

建议

  • React 或 Next.js (首选)
  • 利用现有 AG-UI 生态组件 (如 CopilotKit)

组件库选择 (聊天界面)

  • 核心需求: 多种消息类型展示、流式文本、工具调用、思考状态

通用组件库 (Element UI/Ant Design)

  • 优势: 组件丰富、样式统一
  • 劣势: 无聊天专用组件，需大量自定义

Vercel AI SDK UI

  • 优势: React Hook/组件，考虑流式文本
  • 劣势: 需适配 AG-UI，内置支持有限

CopilotKit React Components

  • 优势: 专为 AG-UI 协议设计，内置解析/渲染逻辑，加速开发
  • 劣势: 可能不如通用库全面，定制性受限

建议

  • 优先 CopilotKit React Components
  • 结合通用库或 Vercel AI SDK UI，自定义消息组件

核心功能实现

  • 流式文本、工具调用、模型思考状态展示
  • 前端状态/UI 映射

流式文本 (TEXT_MESSAGE_CONTENT)

  • START: 创建新消息
  • CONTENT: 追加文本
  • END: 标记完成
  • UI: 实时更新内容

工具调用 (TOOL_CALL_START/ARGS/END)

  • START: 创建工具调用消息，初始化状态
  • ARGS: 追加参数
  • END: 标记完成
  • UI: 展示名称、参数、状态，可能含结果

模型思考状态 (RUN_STARTED/FINISHED/ERROR, STEP_STARTED/FINISHED)

  • RUN_STARTED: 全局加载指示器
  • STEP_STARTED/FINISHED: 更新步骤列表
  • RUN_FINISHED/ERROR: 移除指示器，显示结果/错误

共享可变状态 (STATE_SNAPSHOT/DELTA)

  • SNAPSHOT: 完全替换状态
  • DELTA: JSON Patch 应用增量更新
  • UI: 在侧边栏/独立面板实时渲染

事件流接收、解析与状态同步

  • 关键环节：确保前后端状态实时一致性

事件流接收

  • Fetch API + Web Streams API
  • 基于 SSE 实现流式传输

事件流解析

  • NDJSON (Newline Delimited JSON)
  • 推荐: AG-UI TypeScript SDK (内置解析和事件分发)

前端状态同步

  • 状态管理库 (Zustand, Redux Toolkit, Pinia)
  • 应用 STATE_DELTA (JSON Patch 库如 fast-json-patch + Immer)
  • 错误处理与重同步

快速原型开发步骤 (React + AG-UI SDK + Zustand)

  • 从零开始搭建核心功能

准备工作

  • 安装 Node.js, npm
  • 创建 React 项目 (Vite/Next.js)
  • 安装依赖: @ag-ui/client, @ag-ui/core, zustand, immer, fast-json-patch
  • 准备 AG-UI Agent 后端

设计前端状态结构

  • ChatMessage[] (包含文本、工具、思考等类型)
  • AgentRunState (Agent 运行状态)
  • sharedState (共享可变状态)

实现聊天组件

  • Chat.tsx: 展示消息列表、输入框
  • 使用 useChatStore 交互
  • handleSendMessage: 调用 agent.run().subscribe()

事件流机制：JSON over HTTP/Binary

  • 单一序列的 JSON 事件流
  • 前端通过单一 POST 请求监听流
  • 统一所有通信类型（文本、工具、状态、生命周期）

对前端实时同步和状态管理的影响

  • 实时同步 (打字机效果、工具即时通知)
  • 事件驱动的状态更新
  • 增量状态管理 (JSON Patch)

传输层

  • 标准 HTTP (SSE)
  • 可选二进制序列化器 (@ag-ui/proto)

事件结构

  • 每个事件是 JSON 对象
  • 必需 type 字段（判别器）
  • 最小化载荷 (minimal payload)
  • 示例: TEXT_MESSAGE_CONTENT, TOOL_CALL_START, STATE_DELTA

事件序列性

  • 协议保证事件顺序性
  • 前端无需担心乱序问题

原型进阶

  • 完善消息渲染 (Markdown, 工具结果)
  • 实现共享状态 UI
  • 错误处理与用户反馈
  • 取消功能、并发处理、认证安全、性能优化

集成与样式

  • App.tsx 渲染 Chat 组件
  • App.css 添加基本样式

运行原型

  • 启动 Agent 后端
  • 启动前端开发服务器 (npm run dev)
