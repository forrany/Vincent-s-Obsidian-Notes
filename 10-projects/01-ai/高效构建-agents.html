构建块: 增强型 LLM (Augmented LLM)

代理系统的基础单元，是增强了检索、工具和记忆等能力的 LLM。

  • 关键: 根据用例定制能力，并为 LLM 提供简单、文档化的接口。
  • 实现: 可通过模型上下文协议 (MCP) 等方式。

构建高效 AI 代理

(Building Effective AI Agents)

来源: Building Effective AI Agents \ Anthropic

框架 (Frameworks)

  • 示例: LangGraph, Bedrock AI Agent, Rivet, Vellum.
  • 优点: 简化入门（调用 LLM、定义工具、链接调用）。
  • 缺点: 可能增加抽象层，使调试困难；易诱导过度复杂化。
  • 建议: 从直接使用 LLM API 开始；若使用框架，需理解底层代码。

核心原则 (Core Principles)

  1. 简洁性 (Simplicity): 保持代理设计的简洁。
  2. 透明度 (Transparency): 明确展示代理的规划步骤。
  3. ACI (Agent-Computer Interface): 通过彻底的工具文档和测试，精心制作代理-计算机接口。

构建块、工作流与代理 (Building Blocks, Workflows, and Agents)

何时使用 (When to Use)

  • 简单优先: 尽可能寻找最简单的解决方案。
  • 权衡: 代理系统通常用延迟和成本换取更好的任务性能。
  • 选择:
    • 工作流: 适用于定义明确的任务，提供可预测性和一致性。
    • 代理: 适用于需要大规模灵活性和模型驱动决策的情况。
  • 通常足够: 优化单一 LLM 调用（结合检索和上下文示例）往往已足够。

AI 领域，Agent 本身可以有多种定义，我们更具体的定义为以下几种：

  • 代理系统 (Agentic Systems): 涵盖所有使用 LLM 和工具完成任务的系统。
  • 工作流 (Workflows): LLM 和工具通过预定义的代码路径进行编排。
  • 代理 (Agents): LLM 动态指导自身流程和工具使用，保持对任务完成方式的控制。
Agent 定义

工作流: 提示链 (Prompt Chaining)

将任务分解为一系列步骤，每个 LLM 调用处理前一个的输出。

  • 适用: 任务易于分解为固定子任务。
  • 目标: 通过简化每个 LLM 任务，用延迟换取更高准确性。
  • 示例: 生成营销文案 -> 翻译；撰写大纲 -> 检查 -> 撰写文档。

组合与定制模式 (Combining and Customizing)

  • 这些构建块是常见的模式，可根据用例塑造和组合。
  • 成功的关键是衡量性能并迭代实现。
  • 仅在能显著改善结果时才增加复杂性。

工作流: 评估器-优化器 (Evaluator-Optimizer)

一个 LLM 生成响应，另一个在循环中提供评估和反馈。

  • 适用: 有明确评估标准，且迭代改进能提供可衡量价值。
  • 前提: 人类反馈能改进 LLM 响应；LLM 能提供此类反馈。
  • 示例: 文学翻译（捕捉细微差别）；复杂搜索（多轮搜索与分析）。

自主代理 (Agents - Autonomous)

LLM 在循环中基于环境反馈（如工具调用结果）使用工具。

  • 启动: 用户命令或交互式讨论。
  • 操作: 独立规划和操作...
  • 关键: 每步从环境获取“地面实况”；精心设计工具集和文档 (ACI)。
  • 适用: 开放式问题...
  • 考虑: 成本高，可能出现复合错误...
  • 示例: SWE-bench 编码代理；计算机使用代理。

工作流: 路由 (Routing)

对输入进行分类，并将其导向专门的后续任务。

  • 适用: 复杂任务中存在明显不同类别，且分类准确。
  • 优点: 关注点分离，构建更专业的提示。
  • 示例: 不同客服查询类型路由；根据难易度路由到不同模型 (Haiku vs Sonnet)。

工作流: 并行化 (Parallelization)

LLM 同时处理任务的不同部分，然后聚合输出。

  • 类型:
    • 分段 (Sectioning): 任务分解为独立子任务并行运行。
    • 投票 (Voting): 同一任务运行多次以获得多样化输出。
  • 适用: 子任务可并行以提速；需要多视角或多次尝试以提高置信度。
  • 示例: (分段) 护栏检查与响应分离；(投票) 代码漏洞审查。

工作流: 编排器-工作者 (Orchestrator-Workers)

中心 LLM 动态分解任务，委托给工作者 LLM，并综合结果。

  • 适用: 复杂任务，无法预测所需子任务。
  • 关键区别 (与并行化): 灵活性，子任务由编排器根据输入动态确定。
  • 示例: 涉及多文件修改的编码任务；多源信息收集与分析的搜索任务。

B. 编码代理 (Coding Agents)

从代码补全发展到自主解决问题。

  • 适合原因:
    • 代码解决方案可验证。
    • 代理可利用反馈迭代。
    • 问题空间结构化。
    • 输出质量可衡量。
  • 注意: 人类审查仍重要。

A. 客户支持 (Customer Support)

将聊天机器人界面与工具集成增强能力相结合。

  • 适合原因:
    • 交互自然遵循对话流程...
    • 工具可集成以拉取客户数据...
    • 可编程处理退款...
    • 成功可明确衡量。

工具的重要性

  • 工具使 Claude 能与外部服务和 API 交互。
  • 工具定义和规范应与整体提示一样受到重视。
  • 不同的工具格式对 LLM 的编写难度不同...

工具格式建议

  • 给模型足够的 token “思考”。
  • 保持格式接近模型在网络文本中自然看到的格式。
  • 避免格式化“开销”（如精确计数行数、字符串转义代码）。

ACI (代理-计算机接口) 构建

  • 换位思考: 从模型角度看工具是否易用？描述、参数是否清晰？（提供示例、边缘情况、格式要求、与其他工具的界限）
  • 优化名称/描述: 如同为初级开发者写文档字符串。
  • 测试: 在工作台中运行示例，观察模型错误并迭代。
  • 防错 (Poka-yoke): 修改参数使之更难出错（例如，绝对路径 vs 相对路径）。
